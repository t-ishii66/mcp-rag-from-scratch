# RAG 解説

## RAG とは何か

**RAG（Retrieval-Augmented Generation）** は、LLM に「知らない情報」を答えさせるための手法です。

LLM は学習データに含まれない情報を知りません。社内文書、最新のニュース、独自のデータベースなどは、LLM に直接聞いても正しい答えは返ってきません。RAG はこの問題を解決します。

### RAG の基本的な考え方

```
普通の LLM:
  ユーザー: 「大堀翔の成績は？」
  LLM:     「大堀翔という選手は知りません」（学習データにないため）

RAG を使った LLM:
  ユーザー: 「大堀翔の成績は？」
  → まず関連文書を検索して取得
  → 検索結果を LLM に渡す
  LLM:     「大堀翔の2021年の成績は打率.310、本塁打32本です」
```

RAG は **R**etrieval（検索）+ **A**ugmented（強化された）+ **G**eneration（生成）の略です。

- **Retrieval**: 質問に関連する文書をデータベースから検索する
- **Augmented**: 検索結果を LLM への入力に追加する
- **Generation**: LLM が検索結果を参考にして回答を生成する

## なぜベクトル検索を使うのか

### キーワード検索の限界

普通のキーワード検索（全文検索）では「意味」を理解できません。

```
質問: 「二刀流の選手は？」
文書: 「大堀翔は投手としても打者としても活躍した」

→ 「二刀流」という単語が文書に含まれていないとヒットしない
```

### ベクトル検索なら意味で探せる

ベクトル検索はテキストを数値の配列（ベクトル）に変換し、意味的な近さで検索します。

```
「二刀流の選手」 → [0.12, -0.34, 0.56, ...]  ← 質問のベクトル
「投手としても打者としても活躍」 → [0.11, -0.33, 0.55, ...]  ← 文書のベクトル

→ ベクトルが近い = 意味が似ている → ヒットする！
```

単語の一致ではなく**意味の近さ**で検索できるのがベクトル検索の強みです。

## 埋め込み（Embedding）とは

テキストをベクトルに変換する処理を**埋め込み（Embedding）**と呼びます。

埋め込みを行うには専用のモデルが必要です。
このプロジェクトでは **E5**（EmbEddings from bidirEctional Encoder rEpresentations）という埋め込みモデルファミリーの中から、`intfloat/multilingual-e5-small` を使っています。

- **多言語対応** — 日本語を含む 100 以上の言語に対応しており、日本語テキストの検索精度が高い
- **軽量** — モデルサイズが約 470MB と小さく、CPU でも実用的な速度で動作する
- **Hugging Face で公開** — 初回実行時に自動でダウンロードされ、2回目以降はキャッシュが使われる

```python
# 埋め込みのイメージ
"大堀翔は野球選手です" → [0.12, -0.34, 0.56, 0.78, ..., -0.23]  # 384次元のベクトル
"大堀は投手として活躍"  → [0.11, -0.33, 0.55, 0.77, ..., -0.22]  # 似た意味 → 近いベクトル
"今日の天気は晴れです" → [0.91, 0.12, -0.67, 0.04, ..., 0.88]   # 違う意味 → 遠いベクトル
```

### E5 モデルの特徴

E5 系のモデルにはプレフィックス（接頭辞）の規約があります:

- 文書を保存するとき: `"passage: "` を先頭に付ける
- 検索クエリのとき: `"query: "` を先頭に付ける

```python
# indexer.py での文書保存
E5_DOCUMENT_PREFIX = "passage: "
collection.add(documents=[f"{E5_DOCUMENT_PREFIX}{chunk}"])

# main.py での検索
E5_QUERY_PREFIX = "query: "
collection.query(query_texts=[f"{E5_QUERY_PREFIX}{query}"])
```

このプレフィックスによって、「検索したい意図」と「保存された文書」が正しく対応づけられます。

## ChromaDB（ベクトルデータベース）

[ChromaDB](https://www.trychroma.com/) はベクトル検索に特化したデータベースです。
このプロジェクトでは以下のように使っています:

### データの保存

```python
# ChromaDB にテキストを保存
collection.add(
    ids=["profile.txt_0"],                    # 一意のID
    documents=["passage: 大堀翔は野球選手"],     # テキスト（自動でベクトル化される）
    metadatas=[{"source": "profile.txt"}],     # メタデータ（出典情報）
)
```

`collection.add()` を呼ぶと、ChromaDB が内部で:
1. テキストを埋め込みモデルでベクトル化
2. ベクトルとテキストとメタデータをセットで保存

### データの検索

```python
# ChromaDB で類似テキストを検索
results = collection.query(
    query_texts=["query: 大堀翔の成績"],  # 検索クエリ
    n_results=5,                          # 上位5件
)
```

`collection.query()` を呼ぶと、ChromaDB が内部で:
1. クエリをベクトル化
2. 保存済みの全ベクトルとの距離を計算
3. 距離が近い順にソートして上位 N 件を返す

### 検索結果の構造

```python
results = {
    "documents": [["大堀翔の詳細成績...", "大堀翔のプロフィール..."]],
    "metadatas": [[{"source": "stats.txt"}, {"source": "profile.txt"}]],
    "distances": [[1.73, 1.80]],
    "ids":       [["stats.txt_0", "profile.txt_0"]],
}
```

- `documents`: ヒットしたテキスト本文
- `metadatas`: メタデータ（出典ファイル名など）
- `distances`: クエリとの距離（小さいほど似ている）
- `ids`: ドキュメントID

距離の値は小さいほど似ていることを意味します。ChromaDB のデフォルトは L2（ユークリッド距離の二乗）で、値の範囲は 0 以上です。

検索で返ってくるのは、保存時に登録したチャンク（テキスト断片）そのものです。元のファイル全体や前後のチャンクは含まれません。このため、チャンクの分割サイズが検索結果の情報量に直接影響します。

## チャンク分割

テキストファイルはそのままでは長すぎるため、小さな断片（チャンク）に分割します。

### なぜ分割するのか

```
長い文書をそのまま保存すると:
  - ベクトルが文書全体の「平均的な意味」になってしまう
  - 特定のトピックに対する類似度が下がる
  - 検索精度が悪くなる

チャンクに分割すると:
  - 各チャンクが特定のトピックに集中する
  - ベクトルがそのトピックをよく表現する
  - 検索精度が上がる
```

### チャンク分割の仕組み

このプロジェクトでは **400文字**ごとに分割し、**80文字のオーバーラップ**を設けています。

```
元のテキスト（1000文字）:
[==========================================================]

チャンク1（0〜400文字）:
[===============]

チャンク2（320〜720文字）:   ← 80文字ぶん前のチャンクと重なる
         [===============]

チャンク3（640〜1000文字）:
                  [===============]
```

### なぜオーバーラップが必要か

チャンクの境界で文脈が途切れるのを防ぐためです。

```
オーバーラップなし:
  チャンク1: 「...大堀翔は2021年に打率」
  チャンク2: 「.310を記録した。本塁打は...」
  → 「打率.310」が2つのチャンクに分断される

オーバーラップあり:
  チャンク1: 「...大堀翔は2021年に打率.310を記録した。」
  チャンク2: 「打率.310を記録した。本塁打は32本で...」
  → 両方のチャンクに完全な情報が含まれる
```

### コードでの実装

```python
def split_into_chunks(text: str, chunk_size: int, overlap: int) -> list[str]:
    if len(text) <= chunk_size:
        return [text]  # 短ければ分割不要

    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunk = text[start:end]
        chunks.append(chunk)
        start = end - overlap  # オーバーラップぶん戻る
    return chunks
```

## RAG の全体フロー

このプロジェクトでの RAG の流れを時系列で追います。

### 事前準備（インデックス作成）

```
テキストファイル     チャンク分割        ベクトル化          ChromaDB
profile.txt    →  [チャンク1]  →  [0.12, -0.34, ...] → 保存
               →  [チャンク2]  →  [0.45, 0.12, ...]  → 保存
stats.txt      →  [チャンク1]  →  [0.78, -0.23, ...] → 保存
               →  ...
```

### 質問時（検索 + 回答生成）

```
1. ユーザー: 「大堀翔の成績は？」

2. LLM が「search ツールを使おう」と判断
   → query="大堀翔の成績" で MCP サーバーに依頼

3. MCP サーバーが ChromaDB を検索
   → クエリをベクトル化: [0.13, -0.33, ...]
   → 全チャンクとの距離を計算
   → 距離が近い上位5件を返す

4. 検索結果:
   [1] 出典: stats.txt (類似度: 0.91)
   「大堀翔の詳細成績データ。2021年: 打率.310、本塁打32本...」

   [2] 出典: pro_career.txt (類似度: 0.85)
   「2021年は打撃でもブレイクし...」

5. LLM が検索結果を読んで回答を生成:
   「大堀翔の2021年の成績は打率.310、本塁打32本、
    打点89の好成績を残しました。」
```

## MCP と RAG の関係

RAG 自体は MCP がなくても実現できます。しかし MCP を使うことで:

- **ツールの標準化**: 検索機能を MCP ツールとして定義すれば、どの LLM からでも同じインターフェースで使える
- **分離**: 検索ロジック（サーバー）と対話ロジック（クライアント）を別プロセスに分離できる
- **拡張性**: 別の MCP ツール（計算、API 呼び出しなど）を追加するのが簡単

このプロジェクトでは「MCP のツール呼び出し」として RAG を実現しています。

## まとめ

| 概念 | 説明 |
|------|------|
| **RAG** | 検索結果を LLM に渡して回答を生成する手法 |
| **ベクトル検索** | テキストを数値ベクトルに変換し、意味の近さで検索する |
| **埋め込み** | テキストをベクトルに変換する処理 |
| **ChromaDB** | ベクトル検索に特化したデータベース |
| **チャンク分割** | 長いテキストを小さな断片に分割する。検索精度を上げるため |
| **オーバーラップ** | チャンク間で一部を重複させ、文脈の途切れを防ぐ |
| **E5 プレフィックス** | `"passage: "` / `"query: "` を付けて検索精度を向上させる |
