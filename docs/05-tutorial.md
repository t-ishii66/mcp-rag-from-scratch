# チュートリアル

このチュートリアルでは、プロジェクトを実際に動かしながら MCP と RAG の動作を体験します。

## 前提

[環境構築ガイド](./01-setup.md) の手順が完了していること（`uv sync` まで実行済み）。

---

## ステップ 1: ChromaDB にデータを入れる

まず、テキストファイルをベクトル化して ChromaDB に保存します。

```bash
PYTHONPATH=src uv run python -m server.indexer
```

出力を確認してください:

```
=== インデックス作成開始 ===
...
[1] テキストファイル読み込み
  読み込み: profile.txt (1234文字)
  読み込み: stats.txt (2345文字)
  ...

[2] ChromaDB初期化
  コレクション 'documents' を作成

[3] チャンク分割とインデックス作成
  profile.txt: 4チャンク
  stats.txt: 8チャンク
  ...

=== 完了: 42チャンクをインデックス ===
```

**何が起きたか:**

1. `data/documents/` 内の7つのテキストファイルを読み込んだ
2. 各ファイルを400文字ごとのチャンクに分割した
3. 各チャンクを埋め込みモデル (`intfloat/multilingual-e5-small`) でベクトル化した
4. ベクトルとテキストを ChromaDB に保存した

`chroma_db/` フォルダが生成されていれば成功です。

---

## ステップ 2: ChromaDB の検索を直接試す

MCP を介さず、ChromaDB の検索だけを試せる対話ツールがあります。

```bash
uv run python tools/query_chromadb.py
```

`クエリ>` と表示されたら、検索語を入力します:

```
クエリ> 大堀翔の成績
```

検索結果が距離（distance）付きで表示されます。上位件数を指定することもできます:

```
クエリ> 大堀翔の成績 /3
```

いくつかのクエリを試して、ベクトル検索の動作を確認してください:

- `大堀翔の成績` — 成績データがヒットする
- `トレーニング方法` — トレーニングに関する文書がヒットする
- `年俸` — 給与情報がヒットする
- `こんにちは` — 関連度が低い結果が返る（ベクトル検索は必ず結果を返す）

`quit` で終了します。

**ポイント:** キーワードが文書に含まれていなくても、意味が近ければヒットします。これがベクトル検索の強みです。

---

## ステップ 3: モック LLM でクライアントを動かす

MCP クライアントを起動します。Ollama がなくても動作するモック LLM モードです。

```bash
PYTHONPATH=src uv run python -m client.main
```

起動メッセージを確認:

```
=== MCP RAG クライアント ===
LLMモード: モック
終了するには 'quit' と入力してください

[1] MCPサーバーに接続しました
[2] MCPサーバーからツール一覧を取得: ['search']
[3] ツール定義をLLM用の形式に変換しました

※ 以降、毎回の質問でLLMにツール一覧を渡します
```

ここで何が起きているか:

1. クライアントが MCP サーバーを子プロセスとして起動した
2. JSON-RPC で `initialize` を送り、接続を確立した
3. JSON-RPC で `tools/list` を送り、`search` ツールの定義を取得した
4. ツール定義を Ollama 形式に変換した（モックでも同じ形式を使う）

### ツールが呼ばれるケース

```
質問: 大堀翔の成績を教えて
```

出力:

```
  → LLMに質問とツール一覧を送信...
  → ツール呼び出し: search({"query": "大堀翔の成績を教えて"})
  → 検索結果を取得しました
  → LLMに検索結果を渡して回答を生成中...

【回答】
【モック回答】検索結果を元にお答えします。
...
```

**流れ:**
1. モック LLM が「大堀」というキーワードを検出し、search ツール呼び出しを返した
2. クライアントが MCP サーバーに `tools/call` を送り、検索を実行した
3. 検索結果をモック LLM に渡して回答を生成した

### ツールが呼ばれないケース

```
質問: 今日の天気は？
```

出力:

```
  → LLMに質問とツール一覧を送信...

【回答】
すみません、その質問にはお答えできません。大堀翔選手について聞いてください。
```

**流れ:**
1. モック LLM が検索キーワードを検出しなかった
2. ツール呼び出しなしで直接テキストを返した

`quit` で終了します。

## ステップ 4: デバッグモードで JSON-RPC を観察する

MCP プロトコルの核心部分を見てみましょう。`RAG_DEBUG=1` を付けて起動します。

```bash
RAG_DEBUG=1 PYTHONPATH=src uv run python -m client.main
```

質問を入力すると、JSON-RPC のメッセージが表示されます:

```
質問: 大堀翔の成績

[JSON-RPC] → 送信: tools/call
{
  "method": "tools/call",
  "params": {
    "name": "search",
    "arguments": {
      "query": "大堀翔の成績"
    }
  }
}

[JSON-RPC] ← 受信: tools/call
{
  "content": [
    {
      "type": "text",
      "text": "[1] 出典: stats.txt (類似度: 0.91)\npassage: 大堀翔の詳細成績データ..."
    }
  ]
}
```

**注目ポイント:**

- `→ 送信`: クライアントからサーバーへのリクエスト
- `← 受信`: サーバーからクライアントへのレスポンス
- `method`: 呼び出すメソッド名（`tools/call`）
- `params`: メソッドの引数（ツール名と検索クエリ）
- `content`: 検索結果（テキスト形式）

これが MCP プロトコルの通信です。クライアントとサーバーは JSON をやりとりしているだけです。

## ステップ 5: Ollama で本格的に動かす（任意）

Ollama がインストール済みの場合、実際の LLM を使って動かせます。
まだインストールしていない場合は [環境構築ガイドの「Ollama を使う場合」](./01-setup.md#ollama-を使う場合任意) を参照してください。

```bash
# Ollama が起動していることを確認
ollama list

# Ollama モードで起動
USE_OLLAMA=1 PYTHONPATH=src uv run python -m client.main
```

モック LLM との違い:

| | モック LLM | Ollama |
|---|-----------|--------|
| ツール判定 | キーワードマッチ | LLM が意味を理解して判断 |
| 回答生成 | テンプレート | 検索結果を読んで自然な回答 |
| 速度 | 瞬時 | 数秒〜数十秒 |

Ollama モードで試してほしいこと:

```
質問: 大堀翔はどんな投手？
質問: 二刀流のトレーニング方法を教えて
質問: 大堀翔の年俸の推移は？
質問: こんにちは（← ツールを使わず直接回答する）
```

## ステップ 6: テストを実行する

プロジェクトのテストを動かして、各コンポーネントが正しく動作していることを確認します。

```bash
uv run pytest -v
```

テストの内容:

| テストファイル | テスト対象 |
|--------------|-----------|
| `test_chromadb.py` | ChromaDB の基本操作（追加・検索） |
| `test_indexer.py` | チャンク分割、ファイル読み込み |
| `test_mcp_server.py` | search ツールの動作、ツール定義 |
| `test_client.py` | モック LLM、ツール形式変換 |
| `test_ollama_tools.py` | モック LLM のツール呼び出し判定 |
| `test_integration.py` | MCP サーバー起動 → 接続 → 検索の統合テスト |

特に `test_integration.py` は、実際に MCP サーバーを子プロセスとして起動し、JSON-RPC で通信する統合テストです。テストコードを読むと MCP の使い方がよく分かります。

## ステップ 7: 自分のデータで試す

`data/documents/` のテキストファイルを差し替えれば、別のトピックの検索ツールになります。

### 7-1. テキストファイルを用意する

`data/documents/` に `.txt` ファイルを置きます。内容は何でも構いません。

例: 好きな技術のドキュメント、料理のレシピ、旅行記など。

### 7-2. インデックスを再作成

```bash
PYTHONPATH=src uv run python -m server.indexer
```

### 7-3. 検索を試す

```bash
# ChromaDB 直接検索
uv run python tools/query_chromadb.py

# MCP クライアント経由
PYTHONPATH=src uv run python -m client.main
```

### 7-4. サーバーの search ツールの説明を変える（任意）

`src/server/main.py` の `search()` 関数の docstring を、自分のデータに合わせて変更すると、LLM がより適切にツールを使ってくれるようになります。

```python
@mcp.tool()
def search(query: str, top_k: int = 5) -> str:
    """
    ここにツールの説明を書く。
    LLM はこの説明を読んで、ツールを使うかどうか判断する。
    """
```

## まとめ: 全体の流れ

このチュートリアルで体験した流れをまとめます。

```
[事前準備]
  テキストファイル → チャンク分割 → ベクトル化 → ChromaDB に保存

[質問時]
  ユーザーの質問
    ↓
  LLM にツール一覧と一緒に送る
    ↓
  LLM が「search ツールを使う」と判断（or 直接回答）
    ↓
  MCP サーバーに tools/call を送る
    ↓
  ChromaDB でベクトル検索
    ↓
  検索結果を LLM に渡して回答生成
    ↓
  ユーザーに回答を表示
```

この流れが **MCP + RAG** の基本パターンです。

次は [MCP プロトコル解説](./02-mcp-explained.md) や [RAG 解説](./03-rag-explained.md) で、それぞれの技術的な背景を深く理解してみてください。
